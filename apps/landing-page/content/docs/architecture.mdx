Zap Pilot is a `pnpm` + Turborepo monorepo. Five apps and one shared TypeScript package each carry a narrow responsibility — what each does, and what it deliberately does not do, is enumerated below.

<Tabs items={['account-engine', 'alpha-etl', 'analytics-engine', 'intent-engine', 'frontend & landing-page']}>

<Tab>

**Stack.** Hono · port 3004.

**Does.** User accounts, wallet onboarding, Telegram bot wiring, background job dispatch (weekly reports, daily suggestions), API token management.

**Does not.** Sign or dispatch on-chain transactions. Custody keys. Decide allocation.

Backed by Supabase Postgres with two clients: anon (RLS-enforced) by default, service-role only for admin flows.

</Tab>

<Tab>

**Stack.** Express · port 3003.

**Does.** HTTP-triggered ingestion pipelines that pull pool APRs (DeFiLlama), wallet balances (DeBank), Hyperliquid vault APR, the Fear & Greed Index, token prices (CoinGecko), and stock prices into Postgres. Computes derived series like the 200-Day Moving Average.

**Does not.** Decide allocation. Trigger trades. Talk to wallets.

</Tab>

<Tab>

**Stack.** FastAPI (Python 3.11+, `uv`) · port 8001.

**Does.** Read-only `/api/v2` surface for portfolio trends, risk metrics, market sentiment, and backtests. The current rotation strategy lives here — including the regime engine, weight allocator, and the strategy snapshot fixtures used by the marketing site.

**Does not.** Write to the database. The DB connection is read-only by policy.

</Tab>

<Tab>

**Stack.** Shared TypeScript package — `@zapengine/intent-engine`.

**Does.** Detects wallet capabilities (`wallet_getCapabilities`), selects the execution strategy (EIP-7702 → Multicall3 → sequential), encodes the batched calls, and prepares the transaction the user signs. Builders for swap / supply / withdraw / rotate intents.

**Does not.** Custody keys. Submit transactions on the user's behalf. Decide allocation — that is upstream.

Key files: `src/execution/capability.detector.ts`, `src/execution/eip7702.executor.ts`, `src/execution/multicall3.executor.ts`.

</Tab>

<Tab>

**Stack.** `frontend` is React 19 + Vite (port 3000); `landing-page` is Next.js 15 + Fumadocs MDX (port 3000 — don't run them together). Both deploy to Vercel.

**Does.** UI surfaces. The landing page is the marketing site and these docs. The frontend is the dashboard SPA.

**Does not.** Custody keys. Decide allocation. Run cron jobs.

</Tab>

</Tabs>

## Data flow at a glance

1. **alpha-etl** pulls market data (prices, APRs, sentiment) on schedule into Postgres.
2. **analytics-engine** reads from Postgres, evaluates regime signals, and exposes the result as a read-only API.
3. **account-engine** orchestrates user-facing concerns — onboarding, the Telegram bot, scheduled jobs.
4. When a regime shift fires, **intent-engine** turns the target allocation into a single bundled transaction sized to the user's wallet capability.
5. The **frontend** (or Telegram bot) presents the bundle; the user signs from their own EOA.
